## MAJOR (2)

### M1. `retryWhen` re-subscribes to channel backlog — risk of duplicate/replayed processing

**Verification result**:
<span style="display:inline-block; background:#15803d; color:#ffffff; padding:3px 8px; border-radius:4px; font-weight:700;">NOT CONFIRMED</span>
`receiveAsFlow()` consumes from `Channel`; retrying the upstream collection does not replay elements already received from `intentsChannel`.
Buffered intents that were not yet received may still be processed after retry, but that is continuation, not duplicate replay.
For `CONCURRENT` handling, the more plausible risk is cancellation/loss of in-flight sibling handlers when one handler fails, not replay of the channel backlog.

**File**: `core/src/main/java/cc/colorcat/mvi/internal/ReactiveContractImpl.kt:228-232`
**Category**: correctness, error_handling

```kotlin
val snapshots = intentsChannel.receiveAsFlow()
    .toPartialChange(transformer)
    .retryWhen { cause, attempt -> retryPolicy(attempt, cause) }
```

`retryWhen` wraps `toPartialChange` but re-subscribes from the upstream `receiveAsFlow()`. When the handler for intent `X` throws and the retry policy says "retry", **all** buffered intents in `intentsChannel` are replayed — even ones that had already produced `PartialChange`s before the crash. This means:

- If intents A, B, C are buffered, and A's handler throws, retry replays A, B, C again (B and C are double-processed).
- If B's handler is slow and has side effects (network writes, IPC), the side effect fires again.
- If the exception is persistent (e.g. `IllegalArgumentException` from bad input), the 3-retry default replays the buffer 3 times.

**Impact**: Silent duplicate state changes, double side effects, potential infinite retry loops on persistent errors.

**Recommendation**: The pattern is intentional (the pipeline restarts for subsequent intents), but the documentation doesn't acknowledge the replay risk. Mitigations:
1. Make `intentQueueCapacity` tighter so retries replay fewer items.
2. Document explicitly: "intents already in the channel buffer when a handler exception occurs will be replayed on retry".
3. Default retry policy should be more conservative — or at minimum log the count of buffered intents being replayed.

### M2. `debounceLeading` uses `System.currentTimeMillis()` — non-monotonic clock

**Verification result**:
<span style="display:inline-block; background:#2563eb; color:#ffffff; padding:3px 8px; border-radius:4px; font-weight:700;">FIXED</span>
Fixed on 2026-06-14: `debounceLeading` now uses `System.nanoTime()` via an internal nanosecond time source and converts the window with `TimeUnit.MILLISECONDS.toNanos(...)`.
Added `DebounceLeadingTest.debounceLeading uses monotonic nanosecond clock`; verified with `./gradlew :core:test`.

**File**: `core/src/main/java/cc/colorcat/mvi/MviExtensions.kt:138-146`
**Category**: correctness

```kotlin
var time = -timeMillis
collect { value ->
    val prev = time
    time = System.currentTimeMillis()
    if (time - prev >= timeMillis) {
        emit(value)
    }
}
```

`System.currentTimeMillis()` is subject to NTP adjustments, user clock changes, and daylight saving transitions. If the clock jumps **backward** by ≥ `timeMillis`, the delta `time - prev` goes negative, `time - prev >= timeMillis` is `false`, and the **next** event is silently dropped (it never emits). If the clock jumps **forward**, the guard may not fire when it should.

**Impact**: Throttle may miss events after clock adjustments. On mobile, this is low-probability but non-zero (NTP sync on boot, airplane mode recovery).

**Recommendation**: Replace with `System.nanoTime()` — monotonic, immune to wall-clock changes.

```kotlin
var lastEpoch = -timeMillis * 1_000_000L  // nanos
collect { value ->
    val now = System.nanoTime()
    if (now - lastEpoch >= timeMillis * 1_000_000L) {
        lastEpoch = now
        emit(value)
    }
}
```

## MINOR (7)

### m1. `ReactiveContractLazy` is not thread-safe despite `@Volatile`

**Verification result**:
<span style="display:inline-block; background:#d97706; color:#ffffff; padding:3px 8px; border-radius:4px; font-weight:700;">PARTIAL</span>
The implementation is indeed non-atomic (`cached ?: create().also { cached = it }`) and can create multiple contracts under concurrent access.
However, the KDoc explicitly states it is not thread-safe and explains `@Volatile` as visibility-only, so this is an acknowledged design trade-off rather than an undocumented mismatch.

**File**: `core/src/main/java/cc/colorcat/mvi/MviViewModels.kt:226-235`
**Category**: resource_management

```kotlin
@Volatile private var cached: ReactiveContract<I, S, E>? = null
override val value: ReactiveContract<I, S, E>
    get() = cached ?: create().also { cached = it }
```

Two threads calling `value` concurrently both see `null`, both create contracts, but only one is retained. The KDoc explicitly admits this. **Risk**: low — ViewModels are main-thread-only in practice. But the `@Volatile` suggests intent of thread safety while the implementation doesn't deliver it.

**Recommendation**: Either remove `@Volatile` (honest about single-thread-only) or use `synchronized(this) { ... }`.

### m2. `intentsChannel` capacity is hardcoded at 64, not configurable

**Verification result**:
<span style="display:inline-block; background:#d97706; color:#ffffff; padding:3px 8px; border-radius:4px; font-weight:700;">PARTIAL</span>
The intermediate `intentsChannel` capacity is fixed by private `INTENT_BUFFER_CAPACITY = 64` and is not configurable.
The current class/property KDoc already documents this fixed capacity, so only the configurability concern remains.

**File**: `core/src/main/java/cc/colorcat/mvi/internal/ReactiveContractImpl.kt:125,155-158`
**Category**: design

`INTENT_BUFFER_CAPACITY = 64` is a `private const val`. The dispatch channel (`dispatchQueue`) uses the user-configurable `intentQueueCapacity`, but this intermediate buffer is fixed. Users have no way to tune the buffer between dispatch and the pipeline.

**Recommendation**: Document the fixed capacity in the class KDoc or make it a constructor parameter.

### m3. Handler lookup is exact class match — inheritance not supported

**Verification result**:
<span style="display:inline-block; background:#d97706; color:#ffffff; padding:3px 8px; border-radius:4px; font-weight:700;">PARTIAL</span>
The lookup is exact-class based (`handlers[intent.javaClass]`), so base-class or interface registrations do not catch subclass dispatches.
The behavior is already documented in `IntentHandlerRegistry.register` KDoc and falls back with a warning log, so it is a sharp edge rather than a hidden behavior.

**File**: `core/src/main/java/cc/colorcat/mvi/IntentHandlers.kt:247`
**Category**: design

```kotlin
val handler = handlers[intent.javaClass] as IntentHandler<I, S, E>?
```

`intent.javaClass` lookup fails for subclass dispatch. If you register `register(BaseIntent::class.java, ...)` and dispatch `DerivedIntent()`, it falls through to `defaultHandler`. The KDoc calls this out, but it's a sharp edge that will produce silent "no handler" fallback.

**Recommendation**: Add a `require` or `check` in the reified `register<I>` extension to verify `I` is a concrete (non-abstract) class, or implement hierarchy walk for lookup.

### m7. Default retry policy catches `Exception` but recommends against itself

**Verification result**:
<span style="display:inline-block; background:#2563eb; color:#ffffff; padding:3px 8px; border-radius:4px; font-weight:700;">FIXED</span>
Fixed on 2026-06-14: `defaultRetryPolicy` now retries only `IOException` on attempts `0..2`.
KDoc, README, AGENTS, and `KMviTest.retryPolicy default` were updated to reflect that programming/runtime errors are not retried by default.

**File**: `core/src/main/java/cc/colorcat/mvi/KMvi.kt:207-214`
**Category**: design

The default `defaultRetryPolicy` retries all `Exception`s (3 attempts), while its own KDoc warns "Exception is a broad category". The default itself is a footgun — it will retry `IllegalStateException`, `IllegalArgumentException`, `NullPointerException`, none of which are transient.

**Recommendation**: Default should only retry `IOException` (network) and similar transient exceptions. A comment acknowledges this but doesn't fix it.

### s4. `dispatch` double-check pattern

**Verification result**:
<span style="display:inline-block; background:#15803d; color:#ffffff; padding:3px 8px; border-radius:4px; font-weight:700;">NOT CONFIRMED</span>
The `scope.isActive` guard is not simply redundant: after scope cancellation there can be a race window where `dispatchQueue` is not closed yet, so `trySend` could otherwise report success for work that will not be forwarded.
The two checks cover different states.

**File**: `core/src/main/java/cc/colorcat/mvi/internal/ReactiveContractImpl.kt:317-328`

```kotlin
if (!scope.isActive) { logger.w(...); return }
if (dispatchQueue.trySend(intent).isFailure) {
    if (dispatchQueue.isClosedForSend) { ... }
    else { ... }
}
```

The `scope.isActive` check is redundant with `isClosedForSend` check inside `isFailure`. If the scope is inactive but the queue isn't closed yet, `trySend` will return `isFailure` (channel closed). Both paths log warnings. The `scope.isActive` guard is a fast-path optimization but creates two code paths for the same outcome. Minor complexity.

### s5. `IntentTransformer` factory vs fun interface inconsistency

**Verification result**:
<span style="display:inline-block; background:#15803d; color:#ffffff; padding:3px 8px; border-radius:4px; font-weight:700;">NOT CONFIRMED</span>
The companion `internal operator fun invoke(strategy, config, handler)` exists in `IntentTransformer`, and tests already construct transformers through this factory.

The `fun interface IntentTransformer` is constructed with `IntentTransformer(strategy, config, handler)` across the codebase. For a `fun interface`, this must go through a companion `operator fun invoke` — but no such factory is visible in the file. If it doesn't exist, the code shouldn't compile. Verify and document the factory path.

## STRONG AREAS

1. **Thread-safe core**: `ConcurrentHashMap` in `IntentHandlerDelegate`, Channel-based FIFO, `flowOn(IO/Default)` boundaries. The three-queue architecture (dispatch → intents → snapshot) is well-reasoned.

2. **API ergonomics**: `contract() { register<...> { } }` DSL is clean. `asContract()` provides proper read-only encapsulation via `ReadOnlyContract` (not a cast-shy wrapper).

3. **Documentation**: Every public type and function has thorough KDoc with usage examples. The `HandleStrategy` KDoc is excellent — real examples, diagrams, ProGuard warnings.

4. **Test quality**: 124 tests, all passing. Snapshot/PartialChange immutability is well-covered. The `ReactiveContractImplTest.kt` event flow tests correctly use `UnconfinedTestDispatcher` to avoid `WhileSubscribed` timing issues.

5. **Error isolation**: `PartialChange.apply` exceptions are caught inside `scan`, preventing a bad state transformation from killing the pipeline.
